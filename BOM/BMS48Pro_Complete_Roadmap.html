// L610 模块定位策略 — 最大竞赛亮点

L610（移远 4G Cat.1 模块）通过 UART AT指令与G474通信，支持 TCP/MQTT/HTTP。 推荐方案：MQTT接入阿里云IoT / EMQX，实现BMS数据云端存储+手机端实时查看+远程报警短信。 这是普通同学用ESP8266绝对做不到的层级，评委见到"4G + 云平台"必然加分。

方案A：MQTT云监控 ★ 推荐

L610 AT指令建立MQTT连接到阿里云/EMQX，G474定时发布BMS状态JSON。手机App（MQTT Dashboard）实时查看，Qt上位机同时订阅同一Topic。无需服务器，云平台免费额度够用。

方案B：远程报警短信

检测到过压/过温/短路等故障时，G474控制L610发送短信到指定手机号。AT+CMGF=1 + AT+CMGS 两条指令实现，演示时现场触发故障，手机秒收短信，冲击力极强。

方案C：OTA 远程升级

L610 HTTP GET从服务器下载新固件.bin，G474 BootLoader通过UART接收写Flash。实现真正意义上的无线远程升级，是技术扩展1"OTA升级"的4G版本。

最终建议

A + B 组合：平时MQTT上报数据，故障时自动触发短信报警。两个功能共用同一UART，代码量可控，演示效果翻倍。

开发环境搭建 + 基础外设验证 W1 ~ W2  |  约14天

从零建立完整开发环境，打通所有芯片基础通信，是整个项目的地基。不要跳过这一步直接写业务逻辑。

STM32CubeIDE 环境 安装CubeIDE + CubeMX。G474选用HAL库。开启FPU（G474带硬件浮点，SOC计算用得到）。配置HSE 8MHz + PLL → 170MHz系统频率。
I2C1 → BQ76940 通信 CubeMX配置I2C1 PB8/PB9，400kHz Fast Mode。验证：读取BQ76940设备地址(0x18) → 读SYS_STAT寄存器返回0x00即通信正常。先跑通读写，再做业务。
USART3 → 串口调试 配置USART3 115200/8N1，重定向printf到串口。后续所有调试信息走这里。接CH340到PC，用串口助手验证输出。
USART2 → L610 AT握手 配置USART2 115200，DMA接收。发送AT\r\n，L610返回OK即通信正常。再验证AT+CGSN读IMEI，确认模块功能正常。
FDCAN1 基础配置 CubeMX配置FDCAN1，标准帧500Kbps。接TJA1051收发器。用两块板子互发测试帧，或用USB-CAN适配器+PC端软件验证。
GPIO 中断验证 BQ76940 ALERT引脚配置为外部中断（下降沿触发）。模拟触发，验证EXTI回调正常执行。BQ76200 MOS控制GPIO输出测试。

02
BQ76940 采样 + 保护逻辑 W3 ~ W4  |  约14天

BMS核心功能：13节单体电压读取、NTC温度、总电流；过压/欠压/过温/过流/短路五大保护；均衡控制。这是评委最看重的嵌入式核心能力。

BQ76940 寄存器驱动 编写完整BQ76940驱动库：BQ76940_Init()、BQ76940_ReadAllVoltages()、BQ76940_ReadCurrent()、BQ76940_ReadTemp()。13路电压用VC1~VC15寄存器（跳过VC5/VC10/VC15），每路2字节，100μV/LSB。
过压/欠压保护 配置OV_TRIP寄存器（4.2V）、UV_TRIP寄存器（2.8V）。BQ76940硬件自动触发，ALERT引脚拉低→G474中断→读SYS_STAT→控制BQ76200断开MOS。硬件保护响应时间 <1ms，竞赛重点。
过流/短路保护 配置SCD_THRESH（短路，44mV/Rshunt）和OCD_THRESH（过流，22mV/Rshunt）。短路响应时间70μs，这是BQ76940的硬件级保护，演示时直接短路展示。
均衡控制逻辑 读取13节电压 → 找最低电压节 → 其余电压超出阈值(默认10mV)的节写CELLBAL1/2寄存器开启均衡电阻。500ms周期轮询。写入BAL_CTRL寄存器控制内置均衡开关。
BQ76200 主回路控制 GPIO控制BQ76200的CHG/DSG引脚。充电过压→断CHG；放电欠压/过流→断DSG；短路→双断。故障恢复后延迟重启逻辑（防止频繁切换）。
// BQ76940 读13节电压核心逻辑 void BQ76940_ReadAllVoltages(BMS_Data_t *bms) { uint8_t reg = REG_VC1_HI; uint8_t buf[26]; // VC1~VC15 各2字节，跳过5/10/15 HAL_I2C_Mem_Read(&hi2c1, BQ76940_ADDR, reg, 1, buf, 26, 10); int idx = 0; for(int i = 0; i < 15; i++) { if(i==4 || i==9 || i==14) continue; // 跳过VC5/10/15 uint16_t raw = ((buf[i*2]&0x3F)<<8) | buf[i*2+1]; bms->cell_mv[idx++] = raw * 100 / 1000; // 单位mV } bms->min_cell = find_min(bms->cell_mv, 13); bms->max_cell = find_max(bms->cell_mv, 13); bms->delta_mv = bms->max_cell - bms->min_cell; }

03
FreeRTOS 多任务框架 W4 ~ W5  |  约10天（与Phase2并行后期）

将前面的驱动代码整合进FreeRTOS 7个任务。优先级设计是关键——故障保护任务必须最高优先级，L610通信最低，确保实时性。

任务优先级设计 T1故障保护(Pri=7)：ALERT中断信号量触发，<1ms响应断MOS
T2采样任务(Pri=6)：100ms周期，读BQ76940电压电流温度
T3均衡控制(Pri=5)：500ms周期，依赖T2数据
T4 SOC计算(Pri=4)：1s周期，读BQ34Z100
T5 CAN通信(Pri=3)：事件驱动，发送BMS帧
T6 L610 MQTT(Pri=2)：5s周期上报，故障时立即触发
T7辅助(Pri=1)：串口命令解析、看门狗喂狗
任务间数据共享 全局BMS_Data_t g_bms结构体 + osMutex互斥锁保护。T2写，其他任务读。故障标志用osEventFlags，T1写入故障位，T6订阅事件立即发报警。
看门狗集成 IWDG设置3s超时。T7每1s喂狗一次。T1~T6设置心跳标志位，T7检查所有任务心跳，任意任务死锁超2s未更新 → T7停止喂狗 → 系统复位。这是看门狗保护的完整实现。
// FreeRTOS 任务结构 (cmsis_os2 写法) // 故障保护任务 - 最高优先级 void Task_FaultProtect(void *arg) { for(;;) { // 等待ALERT中断信号量，超时100ms if(osSemaphoreAcquire(alertSem, 100) == osOK) { uint8_t stat = BQ76940_ReadSysStat(); if(stat & (OV|UV|SCD|OCD|OT)) { BQ76200_DisconnectAll(); // 断MOS osEventFlagsSet(faultFlags, stat); // 通知L610任务 g_bms.fault_code = stat; } } g_bms.heartbeat[0]++; // 喂看门狗心跳 } } // BMS数据结构 typedef struct { uint16_t cell_mv[13]; // 13节电压 mV int16_t current_ma; // 电流 mA (正=充电) uint8_t soc; // 剩余电量 % uint8_t temp_c; // 温度 ℃ uint16_t cycle_count; // 循环次数 uint8_t fault_code; // 故障位图 uint8_t heartbeat[7]; // 各任务心跳 } BMS_Data_t;

04
BQ34Z100 SOC计算 + 电量管理 W5  |  约5天

BQ34Z100-G1通过I2C提供精确的库仑积分SOC、循环次数、健康度（容量学习）。这是区别于简单电压估算SOC的核心技术亮点。

BQ34Z100 初始化 首次使用需要"学习周期"（满充→完全放电→满充），之后芯片自动校准容量。竞赛前提前完成学习，演示时直接读取精确SOC。I2C地址0xAA，使用标准访问命令。
关键数据读取 CMD_StateOfCharge(0x2C) → SOC%
CMD_RemainingCapacity(0x10) → 剩余容量mAh
CMD_CycleCount(0x2A) → 循环次数
CMD_StateOfHealth(0x2E) → 健康度%
每秒读取一次，存入g_bms结构体
容量学习集成 BQ34Z100自动执行容量学习，每次完整充放电周期后更新FullChargeCapacity。健康度 = 当前满充容量 / 初始额定容量。G474通过I2C读取，通过Qt上位机展示衰减曲线。
✓ 精确SOC读取 ✓ 循环次数统计 ✓ 健康度评估 ✓ 剩余容量mAh

IAP BootLoader + OTA远程升级 W9  |  约7天

BootLoader工程 独立CubeIDE工程，链接脚本设置起始地址0x08000000，大小32KB。上电后等待3秒串口升级命令，超时自动跳转0x08008000执行APP。设置VTOR寄存器。
Ymodem串口IAP BootLoader实现Ymodem接收端：握手→接收文件名包→循环接收数据包→CRC16校验→HAL_FLASH_Program写入0x08008000起。Qt端实现Ymodem发送，进度条显示，完成后自动复位。
